What is Paging?

Paging is a memory management technique that divides a process’s virtual address space into fixed-size pages and divides physical memory into equally sized frames. The operating system maintains a page table to map virtual pages to physical frames.

Why paging exists

Without paging, programs would need a single contiguous block of RAM, which leads to external fragmentation and inefficient memory usage. Paging lets the OS place pages anywhere in physical memory and keep programs isolated from one another.

  • Isolation between processes via per-process page tables
  • Flexibility to load only actively used pages (demand paging)
  • Simpler allocation because frames are fixed size

What is a page fault?

A page fault happens when a program references a virtual page that is not currently in RAM. The OS pauses the process, finds the page on disk (swap/pagefile), loads it into a free frame (or replaces an existing page), updates the page table, and then resumes execution.

Performance note
Disk access is orders of magnitude slower than RAM. A good replacement policy reduces page faults—especially bursts of faults known as thrashing.

Paging workflow (high level)

  1. CPU generates a virtual address (page number + offset).
  2. MMU consults the TLB/page table to translate page → frame.
  3. If present: memory access proceeds (a page hit).
  4. If absent: a page fault occurs; the OS loads the missing page, possibly evicting another page if no free frames remain.
  5. Update metadata (valid bit, frame number, timestamps/usage bits depending on policy).

The three classic replacement algorithms

A page replacement algorithm decides which page to evict when memory is full. Each policy uses a different kind of “signal” (age, predicted future, or recent usage) to make a choice.

FIFO

First In, First Out

FIFO evicts the page that has been in memory the longest. It’s conceptually simple: maintain a queue of pages in the order they were loaded.

  • Strength: easy to implement and fast
  • Weakness: can evict “hot” pages that are still heavily used
  • Quirk: can show Belady’s anomaly (more frames → more faults)
Optimal

Optimal (MIN)

Optimal evicts the page that won’t be used for the longest time in the future. This policy produces the minimum possible number of faults for a given reference string and frame count.

  • Strength: theoretical best (benchmark policy)
  • Weakness: requires knowledge of the future
  • Use: evaluating how close real policies get to the ideal
LRU

Least Recently Used

LRU evicts the page that hasn’t been used for the longest time in the past, based on the idea of temporal locality: pages used recently are likely to be used again soon.

  • Strength: often performs close to optimal in practice
  • Weakness: exact LRU needs tracking (timestamps/stack) and can be costly
  • Reality: many systems approximate LRU with reference bits or aging

A small example intuition

Suppose the next reference is a page that’s not currently in the frames. FIFO will remove the oldest page, even if it’s about to be referenced again. LRU will remove the page you haven’t touched in the longest time, which often aligns with “least useful now”. Optimal would remove the page whose next future use is farthest away (or never occurs again).

The best policy depends on the access pattern. Sequential scans behave differently than looping patterns; working sets change over time; and real systems need policies that are both effective and cheap to maintain.

Here’s a practical way to compare: what data each policy needs, how expensive it is to track, and what kind of behavior it tends to produce.

Real OSes often approximate “recency” because it’s a strong heuristic and can be tracked cheaply using bits/counters rather than exact timestamps.

When page faults become a problem

If the working set of a process (the pages it needs frequently) doesn’t fit in the available frames, the system can spend most of its time swapping pages in and out.

  • Thrashing: very high fault rate; low CPU utilization
  • Symptoms: slow response, constant disk activity, poor throughput
  • Fix: more frames, better replacement, or limit multiprogramming

Replacement algorithms matter because they decide how quickly you converge to keeping the working set in RAM.
